Project (Due Thurs., Sept. 29, 2016, by 11:59 p.m.)

Objective

To understand the Unix/Linux socket API by developing a C client application capable of correctly interacting with servers, using both TCP and UDP.

Specifications

You are to develop a network client application that will display, upon request, the current weather forecast for the cities in which Big Ten football games will be played.  The client will first use a datagram (UDP) socket to communicate with a server (hereafter referred to as the ScheduleServer). The client will send to this server a datagram containing a message of the form <hawkID> <Big Ten University Name>. The ScheduleServer. will respond with a datagram containing a  message of the form <AwayTeam@HomeTeam> <Three-letter Weather Station Code for station closest to the Home Team's location>. The client will open a stream (TCP) connection to the WeatherServer and interact with this server as described below to obtain requested forecast information from the weather station identified by the three-letter code obtained from the SeheduleServer. The ScheduleServer and WeatherServer are both supplied by the instructor.  You need only develop the client.

 

 

The client should establish a TCP connection to the WeatherServer. The WeatherServer is running on host c-lnx000.engr.uiowa.edu and listening on port 23511. After establishing the TCP connection, the client should send the user's hawkID to the WeatherServer (without a terminating Null character of newline character). You MUST use your valid hawkID so that we can log and verify correct operation of your client.
The response of the WeatherServer is:

Hello <hawkID>

For example, suppose your client is called WeatherClient, and is invoked as:

WeatherClient kuhl

The WeatherServer would respond to the initial client message with:

 Hello kuhl

After this, the client should prompt the user to enter the name of a Big 10 University. For example:

 Weather Forecast for which Big Ten University's football game:

The client should send a datagram to the ScheduleServer.  The ScheduleServer will be listening on port number 23510 of the host c-lnx001.engr.uiowa.edu. The message contained in this datagram should be the user's hawkID, followed by a space, followed the name of the University for which the weather station code is sought (no null characters, blanks, line-feeds, or other extraneous characters).  The names of the 14 Big 10 Universities are given in the table below. For those universities whose names are two words--e.g. Ohio State--the space between the two words should be replaced with an underscore (e.g. Ohio_State).  The message should not contain any additional spaces, or other extraneous characters and should NOT be terminated with a Null character ('\0') or a newline character.  An example message might look like:

 

'kuhl Michigan_State'     (the quotes are not part of the message.)

 

The ScheduleServer will respond with a datagram containing next scheduled game for the specified University, followed by a space, followed by the three-letter code for the Weather station closest to the home team's location.. For instance the response to the above query might look like:

 

'Central_Michigan@Michigan_State LAN'   (without the quotes)

 

Note that the string returned by SeheduleServer is NOT terminated with either a null character or a newline character. The client should send a message containing the three-letter weather station code obtained from the Schedule Server to the WeatherServer. For example:

 

'LAN'

The WeatherServer will respond by sending a detailed current forecast for the requested city over the TCP connection.

The client should display the forecast and then prompt the user to enter another Big Ten University name.

If the user enters the string “quit”, the client program should send this string to the server, then close the TCP connection and terminate. It is required that the client send 'quit' correctly so that the server can properly log the transaction. If the client does not "quit" properly then you will lose points.

The program should be written such that the user can query the weather for as many games as desired before terminating the program.  The TCP connection to the WeatherServer should remain open for the duration of the program--i.e. don't close it after each query.

To complete this assignment you will need to hand in (electronically) the source code of your client application, and have a successfully logged transaction with both servers before the due date.
